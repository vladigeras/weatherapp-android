# AGENTS.md - Руководство для работы с проектом weatherapp-android

## Общие принципы

1. Всегда задавайте уточняющие вопросы перед началом работы по задаче
2. При изучении материала по задаче отмечайте проблемы с архитектурой, алгоритмами или неточностями, предлагайте улучшения
3. При реализации функционала всегда пишите покрывающие тесты
4. После завершения работы обновляйте AGENTS.md проекта

## Уроки из текущей сессии

### Проблемы, которые были исправлены:

1. **Гонка условий в WeatherViewModel**:
   - Были добавлены механизмы отмены предыдущих запросов при смене локации
   - Используется переменная `currentJob: Job?` для отслеживания текущего запроса
   - Перед запуском нового запроса предыдущий отменяется

2. **Неправильное обновление состояния в WeatherScreen**:
   - Добавлен `LaunchedEffect` с зависимостями от `savedLatitude` и `savedLongitude`
   - Теперь WeatherScreen корректно реагирует на изменение локации при возврате из экрана выбора локации

3. **Проблемы с передачей данных между экранами**:
   - Исправлена передача координат из LocationSelectionScreen в WeatherScreen через savedStateHandle
   - При выборе локации (как автоматической, так и ручной) координаты теперь правильно сохраняются в savedStateHandle экрана погоды

4. **Добавлены unit-тесты**:
   - Тесты для проверки корректной отмены предыдущих запросов при быстрой смене локаций
   - Тесты для проверки корректного обновления состояния при различных сценариях переключения локаций

### Лучшие практики, реализованные в этом проекте:

1. **Управление корутинами в ViewModel**:
   - Всегда отменяйте предыдущие запросы перед запуском новых
   - Используйте `viewModelScope.launch` с отменой предыдущих job'ов
   - Не используйте `cancelAndJoin()` в `onCleared()` - достаточно простого `cancel()`

2. **Реактивность UI на изменения состояния**:
   - Используйте `LaunchedEffect` с правильными зависимостями для реакции на изменения состояния
   - Не полагайтесь только на начальную инициализация для получения данных

3. **Передача данных между экранами в Navigation Compose**:
   - Используйте `savedStateHandle` для передачи данных между экранами
   - Убедитесь, что принимающий экран реагирует на изменения этих данных

4. **Тестирование асинхронного кода**:
   - Используйте `runTest` из kotlinx.coroutines.test для тестирования корутин
   - Применяйте `advanceTimeBy` для имитации прохождения времени
   - Проверяйте, что только последний запрос влияет на最终 состояние при гонке условий

## Рекомендации для будущей работы

1. При работе с сетевыми запросами всегда учитывайте возможность гонки условий
2. При переходе между экранами проверяйте, что данные корректно передаются и обрабатываются
3. Пишите unit-тесты для критических путей кода, особенно для асинхронных операций
4. Следите за актуальностью зависимостей и версий Kotlin Coroutines
5. При использовании Jetpack Compose обращайте внимание на жизненный цикл composable функций и правильное использование эффектов (LaunchedEffect, SideEffect и т.д.)
